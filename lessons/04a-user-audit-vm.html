Lesson 4a — User Account Audit · with embedded Linux

CIS §5.3 + §5.4 ~15 min 5 controls ← read-only edition
Same content as Lesson 4, with a real Linux VM you can type into. The shell starts with a populated /etc/passwd + /etc/shadow + /etc/login.defs that contain several deliberate findings: a non-root UID-0 account, an empty-password user, a system account with a real shell, and a permissive aging policy. Your job: find them all and fix what should be fixed.

🎯 Why this matters

The first thing any defender does on a new box is "who has access here?" A forgotten service account with a UID-0 alias, an empty-password row in /etc/shadow, a cron user with a valid login shell — these are some of the most common findings in real engagements.

📖 Tools and files used in this lesson

/etc/passwd — the "who is allowed to log in" database. Seven colon-separated fields per line: user:x:UID:GID:GECOS:home:shell. The x in field 2 means "password is in shadow file." Anyone with UID 0 has root privileges; the lesson's first audit finds extras. man 5 passwd

/etc/shadow — the actual hashed passwords plus aging metadata. Root-only readable. Format: user:hash:lastchanged:min:max:warn:inactive:expire:reserved. An empty hash field (::) means no password is required — a critical finding. A leading ! or * means the account is locked. man 5 shadow

/etc/login.defs — system-wide login defaults. Sets the password aging policy (PASS_MAX_DAYS, PASS_MIN_DAYS, PASS_WARN_AGE) for newly created accounts and provides defaults that chage/useradd consult. man 5 login.defs

awk — line-oriented text processor. Fields 1–7 of /etc/passwd become $1…$7 when you set -F:. Patterns like ($3==0) filter rows; the body in braces prints them. man 1 awk · GAWK manual

passwd — change/lock/unlock account passwords. passwd -l user locks by prefixing the password hash with ! (so no password can ever match); passwd -u unlocks. man 1 passwd

usermod — modify an existing account. usermod -s /usr/sbin/nologin user changes the login shell to one that prints a polite message and exits — used to revoke interactive shell access from system/service accounts. man 8 usermod

lastlog — reads /var/log/lastlog and prints each user's most recent login. "Never logged in" entries are normal for system accounts; an unexpected entry on one of those is a finding. man 8 lastlog

last — reads /var/log/wtmp and prints recent login sessions in reverse chronological order. The -a flag puts the source host at the end of each line. man 1 last

🔍 Audit — five quick checks

1. Anyone else with UID 0?

awk -F: '($3==0){print}' /etc/passwd

Expected: exactly one line — root. Anything else is a critical finding.

2. Any empty passwords?

sudo awk -F: '($2==""){print $1}' /etc/shadow

Expected: nothing. Lock any returned accounts: sudo passwd -l <user>.

3. System accounts (UID<1000) with login shells

awk -F: '($3<1000 && $1!="root"){print $1, $7}' /etc/passwd | grep -vE '/(usr/)?sbin/nologin|/bin/false'

Expected: empty (or only special entries you can justify).

4. Password aging policy

grep -E '^PASS_(MAX|MIN|WARN)_' /etc/login.defs

Expected: PASS_MAX_DAYS 365, PASS_MIN_DAYS 1, PASS_WARN_AGE 7.

5. Last-login activity

lastlog | grep -v 'Never logged in'
last -a | head -20

Document anything anomalous in the assessor notes below.

🔧 Remediate — only what's broken

Findings dictate fixes. The lab's deliberate findings + canonical fixes:

# Lock the empty-password account
sudo passwd -l alice

# Change the system account's shell to nologin
sudo usermod -s /usr/sbin/nologin svc-batch

# Apply password aging defaults
sudo sed -i 's/^PASS_MAX_DAYS.*/PASS_MAX_DAYS  365/'  /etc/login.defs
sudo sed -i 's/^PASS_MIN_DAYS.*/PASS_MIN_DAYS  1/'    /etc/login.defs
sudo sed -i 's/^PASS_WARN_AGE.*/PASS_WARN_AGE  7/'    /etc/login.defs

The non-root UID-0 account (backup) is left for you to investigate manually — that's the kind of finding where the right answer depends on context. In a real engagement, you'd document it, ask the customer if it's intentional, then either delete it (userdel backup) or change its UID to a non-zero one.

✅ Verify

Re-run the five audit commands. The locked + nologin findings should go away; the password aging values should match the targets:

sudo awk -F: '($2==""){print $1}' /etc/shadow            # empty
awk -F: '($3<1000 && $1!="root"){print $1, $7}' /etc/passwd \
  | grep -vE '/(usr/)?sbin/nologin|/bin/false'           # only "backup" (intentional finding)
grep -E '^PASS_(MAX|MIN|WARN)_' /etc/login.defs           # 365 / 1 / 7
