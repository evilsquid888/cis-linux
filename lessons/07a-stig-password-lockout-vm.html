Lesson 7a — STIG: Password Complexity + Lockout · with embedded Linux

DISA STIG ~12 min 3 controls ← read-only edition
Same content as Lesson 7, with a real Linux VM you can type into. The lab plants a deliberately-weak /etc/security/pwquality.conf (8-char min, no character-class requirements) and the standard Debian-style /etc/pam.d/common-auth + common-account (no pam_faillock). Your job is to harden both. No stub binaries needed for this one — every command operates on real files.

🎯 Why this matters

Pairs with Lesson 4 (User Account Audit). STIG mandates stricter password rules and an explicit account-lockout policy via PAM. This is one of the clearest "STIG > CIS defaults" examples — same files, just stricter values.

Two configs touch this:

  • /etc/security/pwquality.conf — what counts as a "good enough" new password
  • /etc/pam.d/common-auth + common-account — what happens after N failed logins (faillock)
📖 Tools and concepts used in this lesson

PAM (Pluggable Authentication Modules) — the framework Linux uses to decide "is this login allowed?" Modules are stacked in /etc/pam.d/<service> files. Each line: type (auth/account/password/session) + control (required/sufficient/[default=die]/etc.) + module + arguments. Order matters — auth modules run top to bottom. man 7 PAM

pam_pwquality.so — checks proposed passwords against a quality policy. Configured via /etc/security/pwquality.conf. The credit values work both ways: positive = "free chars off minlen for using this class," negative = "must include at least N of this class." STIG uses -1 across all classes. man 8 pam_pwquality

pam_faillock.so — tracks consecutive auth failures per user, locks the account when threshold is hit. Replaces the older pam_tally2. Two installation points: preauth (check current count before allowing auth) and authfail (increment on failure). man 8 pam_faillock

sed -i '/pat/i ...' — insert a line BEFORE every line matching pat. '/pat/a ...' appends after. The remediate uses both to slot pam_faillock entries on either side of the existing pam_unix line. GNU sed manual

tee -a — append to a root-owned file from a regular shell via stdin redirection. man 1 tee

🔍 Audit

Two greps — pwquality and PAM:

grep -E '^(minlen|ucredit|lcredit|dcredit|ocredit)\s' /etc/security/pwquality.conf
grep -E 'pam_faillock' /etc/pam.d/common-auth /etc/pam.d/common-account

STIG-required values:

  • minlen = 15
  • ucredit = -1 (≥1 uppercase required)
  • lcredit = -1 (≥1 lowercase required)
  • dcredit = -1 (≥1 digit required)
  • ocredit = -1 (≥1 special char required)
  • pam_faillock with deny=3 unlock_time=900 in both common-auth and common-account

🔧 Remediate

sudo apt install -y libpam-pwquality

sudo tee -a /etc/security/pwquality.conf >/dev/null <<'EOF'

# DISA STIG password complexity
minlen = 15
ucredit = -1
lcredit = -1
dcredit = -1
ocredit = -1
EOF

# Faillock — Ubuntu 24.04 ships pam_faillock; enable via direct edits
sudo sed -i '/pam_unix.so/i auth required pam_faillock.so preauth deny=3 unlock_time=900' /etc/pam.d/common-auth
sudo sed -i '/pam_unix.so/a auth [default=die] pam_faillock.so authfail deny=3 unlock_time=900' /etc/pam.d/common-auth
sudo sed -i '/pam_unix.so/i account required pam_faillock.so' /etc/pam.d/common-account
Lab vs production: lockout testing

On a real box you'd verify by creating a throwaway user, SSHing with the wrong password 3 times, watching the 4th attempt get blocked, then unlocking via sudo faillock --user testlock --reset. The lab has no real sshd daemon and no pam_faillock binary, so the active lockout test won't fire. The grep-based verification below still works — it confirms the config files are correct, which is what STIG actually grades.

✅ Verify

Re-run both greps. The pwquality file should show all five STIG values; the PAM files should each have at least one pam_faillock line:

grep -E '^(minlen|ucredit|lcredit|dcredit|ocredit)\s' /etc/security/pwquality.conf
grep -E 'pam_faillock' /etc/pam.d/common-auth /etc/pam.d/common-account
